Mini Redis Overview

Mini Redis 소개

이 프로젝트는 Redis의 핵심 구조를 TCP, RESP, Storage, TTL, Persistence 계층으로 나눠 직접 구현해 보는 mini redis 프로젝트다. 단순한 딕셔너리가 아니라, 네트워크로 명령을 받고 응답을 돌려주는 서버 구조를 만드는 것이 목표다.

한눈에 보기

Redis

메모리 안에 데이터를 저장하고 네트워크로 명령을 받아 처리하는 서버.

TCP

연결 기반 바이트 스트림. 메시지 경계가 없어서 프로토콜 파싱이 필요하다.

RESP

Redis가 쓰는 요청/응답 포맷. 배열과 bulk string을 통해 명령을 명확하게 표현한다.

Event Loop

여러 연결을 받아도 명령 실행은 한 스레드에서 순차 처리하는 구조.

핵심 흐름

클라이언트 -> TCP 연결 -> RESP 요청 bytes -> parser -> ["SET", "foo", "bar"] -> dispatcher -> handler -> storage -> RespValue -> serializer -> RESP 응답 bytes

2번 역할의 핵심

2번은 이 프로젝트의 중심축인 Storage와 Dispatcher를 맡는다.

  • Entry(value, expire_at) 모델을 유지해서 TTL이 자연스럽게 붙도록 설계한다.
  • Dispatcher가 명령어를 알맞은 handler에 연결한다.
  • Storage는 실제 메모리 상태를 관리한다.
  • 만료 여부는 ExpirationManager가 판단하고, 실제 삭제는 Storage가 수행한다.
이번 2번 PR의 핵심은 storage lock을 제거하고, 단일 스레드 event-loop 전제에 맞는 lock-free storage로 정리한 것이다.

SET 요청 예시

["SET", "foo", "bar"]
-> dispatcher.dispatch(...)
-> handle_set(...)
-> storage.set("foo", "bar")
-> Entry(value="bar", expire_at=None)
-> SimpleString("OK")
-> +OK\r\n

TTL 요청 예시

["EXPIRE", "foo", "10"]
-> handle_expire(...)
-> expiration_manager.build_expire_at(10)
-> storage.set_expire_at("foo", expire_at)
-> Integer(1)
["GET", "foo"]
-> storage.get_entry("foo")
-> 만료면 purge
-> 없으면 NullBulkString()
-> 있으면 BulkString(value)

현재 상태를 정확히 말하면

  • core storage, dispatcher, TTL 구조는 상당히 명확하다.
  • snapshot 저장/복구 구조도 있다.
  • 하지만 TCP 서버와 parser는 아직 role1 최종 구현이 반영되기 전 상태다.
  • snapshot autosave는 아직 background thread를 쓰므로, 완전 단일 스레드 구조로는 role4 쪽 추가 정리가 필요하다.
